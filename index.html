01

ليه ده مهم؟

فهم ازاي الـ AI بيقرأ Figma

⚡ الفكرة الأساسية

الـ AI Agent بيتوصل بـ Figma عن طريق بروتوكول اسمه MCP (Model Context Protocol). الـ Agent مش "بيشوف" التصميم زي ما انت بتشوفه — هو بيقرأ بيانات خام: أسماء الـ Layers، الـ Properties، الـ Variables، والـ Auto Layout. لو التصميم مش منظم، الكود هيطلع غلط حتى لو التصميم نفسه تمام.

الـ AI بيشوف ✅ الـ AI مش بيشوف ❌
أسماء الـ Layers والـ Componentsترتيب العناصر visually
الـ Variables (Colors, Spacing, Typography)التأثيرات المعمولة manually
خصائص الـ Auto Layoutاللي عملته بـ absolute position
الـ Variants والـ Propertiesالـ Groups غير المسماة
الـ Constraints والـ Resizingالقصد من التصميم لو مش متوثّق

النتيجة العملية: لما الـ Flutter developer يقول "الـ AI طلّع كود مش صح"، في الغالب المشكلة مش في الـ AI — المشكلة إن الـ Figma file مش بيقدر يوصل المعلومة الكافية. التصميم الصح = كود صح.

04

Figma Variables (Design Tokens)

أهم خطوة — بدونها الكود هيبقى hardcoded بالكامل

⚡ ليه Variables ضرورية؟

لما الـ AI يشوف لون بـ HEX مباشرة زي #FF5733 مش بيعرف هو ده الـ Primary Color ولا اللون ده بس هنا. لما تستخدم Variable باسم brand/primary، الـ AI يفهم إنه لازم يحطه في الـ AppTheme ويستخدمه بالاسم في كل مكان.

❌ بدون Variables
flutter output
// الكود المولّد — hardcoded 😬
Container(
  color: Color(0xFFFF5733),
  padding: EdgeInsets.all(16),
  child: Text(
    'Hello',
    style: TextStyle(
      fontSize: 16,
      fontWeight: FontWeight.w600,
    ),
  ),
)

مستحيل تغير الـ theme لاحقاً بدون ما تدور على كل حاجة

✅ مع Variables
flutter output
// الكود المولّد — theme-aware 🎉
Container(
  color: context.colors.primary,
  padding: AppSpacing.md,
  child: Text(
    'Hello',
    style: context.textTheme.bodyMedium,
  ),
)

تغيير الـ theme من مكان واحد يغير كل حاجة

التصنيفات المطلوبة في الـ Variables

Collection مثال يتحول لـ Flutter يتحول لـ Web
colors/ colors/brand/primary AppColors.primary --color-primary
spacing/ spacing/sm, spacing/md AppSpacing.sm --spacing-sm
typography/ typography/heading/h1 AppTextStyles.h1 --font-heading-h1
radius/ radius/sm, radius/lg AppRadius.lg --radius-lg
elevation/ elevation/card AppElevation.card --shadow-card

Components & Variants

كل عنصر بيتكرر لازم يبقى Component مع Variants للـ states

❌ Group بس

  Rectangle
  Text "Click me"

يولّد كود مكرر في كل شاشة بدون reusable widget

✅ Component مع Variants
⬡ primary_button  ← Figma Component
  Variant: state=default
  Variant: state=loading
  Variant: state=disabled
  Property: label (text)
  Property: icon (boolean)

يولّد widget واحد بيتعامل مع كل الحالات

الـ States المطلوبة لكل Component

🔘

Buttons

default · loading · disabled · pressed · hover

📝

Input Fields

empty · focused · filled · error · disabled

🃏

Cards

default · loading (skeleton) · empty state

📋

List Items

default · selected · disabled · swiped

💡 ليه Variants مهمة؟

الـ Flutter developer لو مش لاقي الـ loading state في التصميم، هيعمله بنفسه وممكن يختلف عن باقي الـ UI. لما الـ AI يشوف الـ Variants، بيولّد enum ButtonState وبيعمل الـ conditional rendering أوتوماتيك.

06

Auto Layout

Auto Layout = Flutter Layout Widgets مباشرةً

⚡ القاعدة الذهبية

ممنوع Absolute Positioning — لما تحط عنصر بـ X/Y ثابتة، الـ AI هيولّد Positioned أو Stack في Flutter وده كود هش جداً ومش responsive. الـ Auto Layout هو اللي بيولّد الـ Column و Row و Wrap الصح.

Figma Auto Layout Flutter Widget CSS/Web
Vertical · Fill Container Column + Expanded flex-direction: column
Horizontal · Fixed Size Row + SizedBox display: flex; width: Xpx
Wrap (Horizontal) Wrap flex-wrap: wrap
Scrollable Vertical SingleChildScrollView overflow-y: auto
Grid Layout GridView display: grid
Absolute Position (Stack) Stack + Positioned ⚠️ position: absolute ⚠️
❌ بدون Auto Layout
// الكود المتوقع
Stack(
  children: [
    Positioned(
      top: 24, left: 16,
      child: Text('Title'),
    ),
    Positioned(
      top: 48, left: 16,
      child: Text('Subtitle'),
    ),
  ]
)

مش Responsive — هيتكسر على شاشات مختلفة

✅ مع Auto Layout
// الكود المتوقع
Column(
  crossAxisAlignment:
    CrossAxisAlignment.start,
  children: [
    Text('Title',
      style: AppTextStyles.h2),
    SizedBox(height: AppSpacing.sm),
    Text('Subtitle',
      style: AppTextStyles.body),
  ],
)

Responsive تلقائياً ✅

المعلومات اللي الـ AI مش يقدر يستنتجها من التصميم لوحده

في Figma، استخدم الـ Dev Mode أو الـ Comments عشان تكتب المعلومات دي على كل شاشة أو Component:

Figma Dev Mode Annotation — product_card
Widget:       ProductCard
State:        BLoC → ProductBloc
Navigation:   GoRouter → /product/:id
API:          GET /api/products/{id}
On tap:       Navigate to ProductDetailScreen
Loading:      Show skeleton variant
Empty:        Show empty_state component
Assets:       product_image → WebP, max 400px
🧭

Navigation

وضّح الـ route لكل زرار أو تاب — مش كل حاجة واضحة من التصميم لوحده

🔌

API Endpoint

إيه الـ API اللي بتجيب البيانات للشاشة دي؟ (GET, POST, ...)

📦

State Type

🖼️

الأيقونات SVG، الصور WebP أو PNG مع الحد الأقصى للأبعاد

08

Checklist التسليم

قبل ما تقول "التصميم اتعمل" — تأكد من كل دول

📐 هيكل الملف

  • صفحات منفصلة Design System · Components · Screens
  • كل الـ Screens مسماة بـ snake_case على صفحة Screens
  • لا توجد Frame أو Group غير مسماة في المستوى الأعلى

🎨 Design Tokens

  • Figma Variables معمولة لـ Colors / Spacing / Typography / Radius
  • لا توجد قيم Hardcoded (HEX أو أرقام مباشرة) على الـ Screens
  • Light & Dark modes معمولين كـ Variable Modes لو في متطلبات لده

⬡ Components

  • كل عنصر بيتكرر محوّل لـ Figma Component (مش Group)
  • كل Component عنده Variants لكل الـ States
  • Component Properties معمولة (text, icon, isDisabled, ...)

📐 Layout

  • Auto Layout مستخدم في كل Frame و Component (لا Absolute Position)
  • Constraints و Resizing محددة صح (Fill / Hug / Fixed)
  • Scroll behavior موضوح على الـ Frames اللي بتتسكرول

📝 Documentation

  • Dev Mode Annotations على كل Screen (Navigation، API، State)
  • Prototype Links موضوحة الـ Navigation Flow
  • Assets exported بالـ format الصح (SVG للأيقونات، WebP للصور)

الخلاصة
الـ AI Agent بيكون بس كويس زي الـ Figma file. التصميم المنظم = كود production-ready.
التصميم الغير منظم = كود hardcoded محتاج refactoring بالكامل.
